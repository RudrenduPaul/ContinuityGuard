# ContinuityGuard

Your short-drama pipeline generates a shot. Did the character actually
stay the same person across cuts? Does that motion look physically real?
ContinuityGuard checks both, on your own machine, before you spend render
credits finding out the hard way.

```bash
npx continuityguard-cli scan ./generated-clips/
```

---

AI short-drama generation is having a real moment, and every title is a
stack of individually generated shots. Generation models still drift: a
character's face shifts slightly between cuts, or a motion jumps in a way
that reads as physically wrong the moment a human watches it. Catching
that after render is expensive. ContinuityGuard scans a folder of
already-generated clips or frames -- from any pipeline, not just one tool
-- and flags the shots worth a second look, before you commit to a
re-render.

## What it does

Real output, from this repo's own committed synthetic fixtures

was available for this build, so every number below is reproducible, not
illustrative):

Every flag includes a timestamp/frame reference, a numeric score, and a
plain-language reason -- so you or your QA reviewer knows exactly why a
shot got flagged, not just that it did. See `CHANGELOG.md` "CG02/CG03
fixture calibration" for the full raw numbers this run's thresholds came
from.

## Install

```bash
npm install -g continuityguard-cli
# or run without installing:
npx continuityguard-cli scan ./generated-clips/
```

## Requirements

- **Node.js >= 22.**
- **A system `ffmpeg` install.** ContinuityGuard checks for it at startup
  and prints the exact install command for your OS if it's missing (e.g.
  `brew install ffmpeg` on macOS, `apt install ffmpeg` on Debian/Ubuntu).
  ContinuityGuard does not bundle a static ffmpeg binary in v0.1 -- a
  bundled per-platform build would be materially larger than this
  project's own dependency footprint and would inherit ffmpeg's own
  shifting LGPL/GPL licensing terms depending on which codecs are
  compiled in. Depending on a system install keeps this package small and
  its licensing surface simple.

No API key, no account, no network call at any point during a scan.

## Usage

```bash
continuityguard scan <directory> [options]

Options:
  --json           print the full machine-readable JSON report to stdout
                    instead of a terminal summary
  --fps <fps>       frame sample rate for ingestion (default: 2.3)
  --out <path>      path to write the JSON report file to
                    (default: ./continuityguard-report.json)
```

```bash
npx continuityguard-cli scan ./generated-clips/
# terminal summary + ./continuityguard-report.json written

npx continuityguard-cli scan ./generated-clips/ --json > report.json
# machine-readable output for a QA agent or CI pipeline to parse
```

### Naming your clips so CG02 can track characters

CG02 infers which character a clip belongs to from its filename, using a
`<character>_<shot-id>.<ext>` convention (e.g. `mei_shot01.mp4`,
`mei_shot02.mp4`). Clips sharing a character prefix are compared against
that character's first-seen shot. There is no industry-standard
character-tagging metadata format across AI short-drama pipelines, so
v0.1 reads it from the filename rather than requiring a separate
manifest. Clips that don't match the convention are still decoded and
scored by CG03, just not compared for character consistency.

## How it is different

ContinuityGuard is not trying to compete with the platforms that generate
this content. Several are deeply funded, well-resourced companies
actively shipping in exactly this space, and any one of them could add a
check like this natively at any time. We're being direct about that
instead of pretending otherwise: this tool's advantage is packaging, not
a technical moat. It works with output from any pipeline, it never
uploads your footage anywhere, and it is free. That's the whole pitch.

## Known limitations (read before trusting a flag)

- **Character-consistency scoring is best-validated on photorealistic
  content.** v0.1 uses a generic ImageNet-pretrained visual-similarity
  embedding (MobileNetV2 -- see `src/score/models/NOTICE.md` for why, and
  why it's not a dedicated face-recognition or CLIP embedding). Its
  accuracy on stylized or anime-adjacent character designs -- which is
  most short-drama content -- is genuinely unverified going into v0.1.
  Treat a flag on stylized footage as "worth a second look," not a
  confirmed defect.
- **The `<character>_<shot-id>` filename convention is a v0.1
  simplification**, not an industry standard. A clip that doesn't follow
  it is scored for physics but not compared for character consistency.
- **Physics-plausibility scoring is a frame-to-frame diff heuristic, not
  optical flow and not a physics simulator.** It flags frame-to-frame
  motion that looks discontinuous relative to a shot's own local
  baseline. It does not "detect" a physics violation, and it will have
  both false positives (legitimate fast motion, stylized jump-cuts) and
  false negatives (subtly implausible motion that stays under the
  threshold).
- **Thresholds are calibrated on a small, fully synthetic fixture set**
  (solid-color clips, not real faces or real motion) -- see
  `CHANGELOG.md` for the exact numbers and the command that produced
  them. They are a real, reproducible starting point, not a
  statistically robust threshold derived from a large labeled dataset.
  Expect them to move as real-world reports come in.
- **Requires a system `ffmpeg` install.** See "Requirements" above.

## Local-only, always

Your unreleased footage never leaves your machine. Every clip is decoded
locally via ffmpeg, every embedding is computed locally via a bundled
offline ONNX model, and the report is written to a local file. No
account, no API key, no upload step, not for a single frame. This is
verified empirically, not just asserted: `npm run verify:zero-network`
monkey-patches every network entry point Node exposes and runs a real
scan against this repo's own fixtures -- if anything tried to reach the
network, that script would fail loudly. It runs in CI on every push. See
`CHANGELOG.md` for the exact result.

## What this is not

- Not a script-to-video generator. ContinuityGuard never generates a
  frame, a clip, or a script -- it only scores footage that already
  exists.
- Not a hosted or cloud service. Every scan runs entirely on your own
  machine.
- Not a paid product. Everything here is free, Apache-2.0 licensed, with
  no paid tier in v0.1.
